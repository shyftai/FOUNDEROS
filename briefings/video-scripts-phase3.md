## Script 3: "Prompt Engineering at Scale: Lessons From Building 200+ Slash Commands"

**Total runtime:** ~4:30 minutes
**Angle:** Behind-the-scenes technical — what makes structured AI workflows work

---

### HOOK (0:00 - 0:10)

[CAMERA] Founder, direct to camera.

"Everyone thinks the secret to AI is the model. GPT-4, Claude, Gemini — pick the smartest one and you win. Wrong. I've built over two hundred structured AI commands, and the model is the least interesting part. The secret is the system prompt architecture."

[TEXT OVERLAY] "Prompt engineering at scale: 200+ slash commands"

---

### STORY / CONTEXT (0:10 - 1:30)

[CAMERA] Founder, storytelling mode.

"Early on, I did what everyone does. I wrote long, clever prompts. Paragraph after paragraph of instructions. 'You are a sales expert. You write concise emails. You always include a clear CTA.' You know the type.

And they worked. Sometimes. When the task was simple and the context was obvious, the output was decent. But the moment things got complex — write a four-touch sequence that follows my brand voice, validates against my ICP, checks my suppression list, and logs the cost of every API call — the long prompt fell apart."

[CAMERA] Slight lean forward.

"The problem wasn't the AI. The problem was me. I was trying to cram an entire operating procedure into one prompt. That's like writing an entire engineering team's workflow in a single README file. It doesn't work.

So I started thinking about prompts differently. Not as instructions — as architecture."

---

### CORE INSIGHT (1:30 - 3:00)

[CAMERA] Founder, teaching mode. Slightly faster pace.

"Let me show you the anatomy of a well-structured slash command."

[SCREEN] A real slash command file, scrolling through sections:

```yaml
# Frontmatter
name: campaign
description: Build a multi-touch outbound campaign
category: execution
requires_approval: true
loads_context:
  - workspace/ICP.md
  - workspace/TONE.md
  - workspace/SUPPRESSION.md
  - workspace/LEARNINGS.md
```

[VO] "First — YAML frontmatter. Not prose. Structured metadata. What does this command do? What context does it need? Does it require human approval? The AI doesn't guess. It reads."

[SCREEN] Next section:

```markdown
## Objective
Build a campaign that matches the workspace ICP,
follows tone of voice rules, and passes all quality gates.

## Execution Context
- Load and parse ICP.md for targeting criteria
- Load TONE.md for voice and style rules
- Check SUPPRESSION.md before any contact inclusion
- Load LEARNINGS.md for historical performance data
```

[VO] "Second — objective and execution context. Not vague instructions like 'write good emails.' Specific, verifiable goals. And explicit context loading — the AI knows exactly which files to read and why."

[SCREEN] Next section:

```markdown
## Process Steps
1. Parse ICP and extract targeting criteria
2. Draft sequence (4 touches, each with distinct angle)
3. Run quality gates:
   - Brief alignment check
   - Merge field validation
   - Suppression list check
   - Spam word scan
   - Readability score
4. Present for approval (APPROVAL mode)
5. Log execution to COSTS.md
```

[VO] "Third — process steps. Not 'figure it out.' A specific, ordered workflow. With quality gates built in. The AI follows a procedure, not a vibe."

[CAMERA] Back to founder.

"This is what I mean by system prompt architecture. Each slash command is a self-contained unit with declared dependencies, structured context loading, explicit quality gates, and a defined approval flow.

The AI model doesn't need to be brilliant. It needs to follow a well-designed process. And when you build two hundred of these, three things happen."

[SCREEN] Three points appear:

```
1. Consistency — every execution follows the same quality bar
2. Composability — commands can call other commands
3. Learnability — the system improves from its own output
```

[CAMERA] Back to founder.

"The output quality stops depending on how clever your prompt is on any given Tuesday. It depends on the architecture. And architecture scales. Cleverness doesn't."

---

## Production Notes

**Filming setup:**
- Talking head: well-lit, clean background, no branded backdrop (founder authenticity, not corporate)
- Camera at eye level, slight shallow depth of field
- Founder should be in casual-professional attire (think "CEO who actually builds things")
- Screen recordings: same dark terminal theme as Phase 1/2 scripts (Dracula or One Dark)

**Pacing and delivery:**
- These are NOT demo videos — they are strategic narratives with proof points
- Hook: punchy, zero warm-up, say something bold immediately
- Story: conversational, first-person, specific details (not generic "we all know...")
- Core insight: teaching mode — slower, more deliberate, let the frameworks land
- Proof: brief and visual — show don't tell, keep it under 60 seconds
- CTA: direct, confident, never desperate

**Voiceover vs. camera balance:**
- 60-70% talking head (camera), 30-40% screen share
- Screen share should support the point, not replace the speaker
- Never stay on screen share for more than 30 seconds without cutting back to camera

**Text overlays:**
- Use sparingly — only for the hook statement and CTA
- Sans-serif, bold, high contrast, lower third
- No bullet point overlays — save that for the screen share sections

**Editing:**
- Jump cuts between camera angles are fine (YouTube style)
- No transitions — hard cuts only
- Light ambient music during hook and CTA, silence during core insight
- Subtle zoom on key screen elements (data points, architecture diagrams)
- Target total runtime: 3:30-4:30 per script (do not pad)

**Platform optimization:**
- YouTube: 1920x1080 horizontal, chapters in description matching script sections
- LinkedIn: same edit, but add burned-in captions (80% of LinkedIn video is watched muted)
- Consider vertical cuts (1080x1920) of hooks only for LinkedIn/Instagram teasers

**Posting cadence:**
- One script per week, Tuesday or Wednesday morning
- LinkedIn post should include a text summary of the core insight (not just "new video")
- YouTube description should include timestamps, repo link, and early access link
