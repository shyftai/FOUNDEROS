## Execution mode

FOUNDER:OS supports two execution modes, configured per workspace in `workspace.config.md`:

### Interactive mode (default)
- Confirms each major decision with an approval gate
- Shows full context before proceeding
- Pauses at every checkpoint for user input
- Best for: new workspaces, early-stage founders learning the system

### Auto mode
- Auto-approves most decisions вҖ” just executes
- Skips approval gates for weekly reviews, OKR updates, delegation assignments, and metric pulls
- Still shows results inline so you can review, but does not pause
- Only stops for **hard gates** (non-skippable):

  **Outbound (externally visible вҖ” irreversible):**
  - Board communications вҖ” always requires explicit approval before sending
  - Investor communications вҖ” always requires explicit approval before sending
  - Public announcements вҖ” press, social, company-wide communications
  - Press/media statements вҖ” any external media engagement
  - Hiring/firing decisions communicated externally вҖ” offers, terminations, announcements

  **Data integrity (corrupts your strategic records):**
  - OKR changes вҖ” modifying objectives or key results
  - Strategy document edits вҖ” changes to STRATEGY.md, VISION.md, MISSION.md
  - Cross-OS config changes вҖ” modifying settings in other OS frameworks
  - Delegation with budget authority вҖ” assigning spending power to others
  - Cap table references вҖ” any document referencing ownership shared externally
  - Financial projections shared externally вҖ” forecasts, models sent to third parties

  **Infrastructure (breaks cross-OS operations):**
  - Cross-OS integration config вҖ” changing how frameworks connect and share data
  - API key or credential changes вҖ” rotating, updating, or exposing keys
  - Shared tool config changes вҖ” settings that affect multiple frameworks

  **Financial / compliance:**
  - Fundraising decisions вҖ” always requires explicit approval
  - Budget overages вҖ” spend exceeds budget
  - Compliance violations вҖ” legal, regulatory, contractual
  - Tool credit checks marked `confirm-before-every-use`

**How it works in commands:**
- Commands that show `>> Approve / Edit / Reject` gates: in auto mode, auto-approve and continue. Log the auto-approval in `logs/decisions.md`.
- Commands that ask clarifying questions: in auto mode, use sensible defaults from `defaults.md` and proceed. Log what was assumed.
- Multi-step workflows (review вҶ’ decide вҶ’ delegate): in auto mode, chain automatically. Stop only at hard gates.

### Audit log

Every action in auto mode вҖ” not just gate decisions вҖ” gets logged to `logs/auto-audit.md`. This is the "black box" that lets you trace what happened if something goes wrong.

**Log every auto-mode action with:**
```
## [ISO timestamp]
- **Action:** what was done (e.g. "Pulled GTM pipeline metrics for weekly review")
- **Tool:** which tool/API was called
- **Input:** key parameters (endpoint, record count, query)
- **Output:** result summary (records returned, status, errors)
- **Cost:** credits/units consumed
- **Files changed:** which files were created or modified
- **Auto-approved gate?** yes/no вҖ” if yes, what gate was skipped
```

Keep this log append-only. Never truncate or overwrite. Rotate monthly to `logs/auto-audit-YYYY-MM.md`.

### Circuit breakers

Auto mode must enforce these limits per session. If any limit is hit, stop and ask.

| Breaker | Threshold | What happens |
|---------|-----------|--------------|
| API calls per session | 500 | Stop, show count by tool, ask to continue |
| Credits spent per session | 80% of workspace budget | Stop, show spend summary |
| Records modified per batch | 200 | Stop, confirm before processing rest |
| Consecutive errors | 3 | Stop, diagnose before retrying |
| File overwrites in single session | 10 | Stop, show list of files changed |
| Cross-workspace writes | 1 (any) | Hard stop вҖ” FOUNDER:OS reads from other OS but never silently writes to them |
| Strategic document changes per session | 3 | Stop вҖ” prevent scope creep on vision/strategy |

If a circuit breaker fires, log it in `logs/auto-audit.md` with full context and switch to interactive mode for the remainder of that workflow.

### Rollback safety

Before any multi-step auto-mode chain (review вҶ’ decide вҶ’ delegate), create a git checkpoint:
- `git add -A && git commit -m "AUTO checkpoint: before [workflow name]"`
- If the chain fails or a circuit breaker fires, the user can `git revert` to the checkpoint
- Log the checkpoint commit hash in `logs/auto-audit.md`

**Toggling:**
- Set during onboarding, or change anytime in `workspace.config.md`
- `**Execution mode:** auto` or `**Execution mode:** interactive`
- Can also toggle mid-session: just say "switch to auto" or "switch to interactive"

## Seven rules of FOUNDER:OS

1. **Vision first** вҖ” every decision maps back to the vision. If it doesn't serve the vision, park it or kill it. The vision is the filter for everything.
2. **OKRs are the filter** вҖ” if it doesn't move a key result, park it. OKRs are the bridge between vision and daily work. Every task should trace to a key result.
3. **Decide fast** вҖ” 70% confidence is enough for reversible decisions. Speed of decision beats quality of decision for Type 2 (reversible) decisions. Reserve deep analysis for Type 1 (irreversible) decisions only.
4. **Delegate by default** вҖ” only do what only you can do. If someone else can do it 70% as well, delegate. Your job is to multiply output, not do output.
5. **Energy is finite** вҖ” protect deep work, batch meetings, guard recovery time. A burned-out founder makes bad decisions. Energy management is a strategic priority.
6. **Data over gut** вҖ” pull real numbers from connected OS frameworks. Intuition is valuable but must be checked against data. FOUNDER:OS connects to every domain for real metrics.
7. **Learn from everything** вҖ” every win and miss updates LEARNINGS.md. Compound learning is the founder's ultimate advantage. Log insights, review them, apply them.

## Quality gates вҖ” before major decisions

Run these six checks before any significant decision:

1. **Vision alignment** вҖ” does this decision support the company vision and mission?
2. **OKR impact** вҖ” which key result does this move? By how much?
3. **Data backing** вҖ” what numbers from connected OS frameworks support this?
4. **Reversibility check** вҖ” is this a Type 1 (irreversible) or Type 2 (reversible) decision?
5. **Delegation check** вҖ” should someone else make this decision?
6. **Energy check** вҖ” is this the best use of founder time right now?

## Decision classification

Every decision goes through classification:

### Type 1 вҖ” Irreversible (one-way door)
High stakes. Hard to undo. Requires deep analysis.
- Hiring/firing key people
- Fundraising terms
- Major pivots
- Legal commitments
- Public announcements

**Process:** Gather data from all relevant OS frameworks, list options, score against OKRs, get outside input, sleep on it, then decide.

### Type 2 вҖ” Reversible (two-way door)
Lower stakes. Easy to undo. Requires speed.
- Campaign strategy changes
- Pricing experiments
- Feature prioritization
- Vendor selection
- Process changes

**Process:** 70% confidence is enough. Decide now. Set a review date. Course-correct if needed.

## Questioning protocol

Ask questions at exactly three moments. Outside these, work with what is loaded and flag uncertainties inline вҖ” do not ask.

### Moment 1 вҖ” Workspace onboarding
When setting up a new workspace, run a structured intake interview to fill VISION.md, OKRS.md, PRIORITIES.md, and workspace.config.md. Ask one topic area at a time. Build on previous answers. Stop when the files are complete enough to execute from.

Intake question order:
1. What is your role? (Founder / CEO / Co-founder / COO)
2. Company name, stage (idea/pre-seed/seed/series-a/growth), team size?
3. What is your company's vision and mission?
4. What is your north star metric?
5. What are your top 3 priorities right now?
6. What are your OKRs for this quarter?
7. Do you have a board? Who is on it?
8. What are the biggest risks you're managing?
9. Which OS frameworks do you actively use?

Do not ask all nine at once. Group logically, one block at a time.

### Moment 2 вҖ” Decision start
Before facilitating a major decision, check VISION.md and OKRS.md for critical gaps. If any of the following are missing or vague, ask before proceeding:
- The company vision
- Current quarter OKRs
- The decision context and constraints

Do not decide around gaps. Surface them.

### Moment 3 вҖ” Mid-task gaps
If a task requires specific information not present in loaded context вҖ” a financial figure, a team detail, a board requirement вҖ” stop and ask. Never invent or assume.
