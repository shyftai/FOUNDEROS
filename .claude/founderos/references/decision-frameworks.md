# Decision Frameworks

Reference for structured decision-making. Load before running `/founder:decide`.

---

## Type 1 vs Type 2 Decisions (Jeff Bezos framework)

### Type 1 — Irreversible (one-way door)
- Hard or impossible to undo
- High stakes and consequences
- Require thorough analysis before committing
- Should involve input from advisors, board, or team

**Examples:**
- Accepting investment terms
- Hiring/firing key executives
- Major pivots
- Signing long-term contracts or leases
- Public announcements that change positioning
- Shutting down a product or business line

**Process:**
1. Gather all available data from connected OS frameworks
2. List all viable options (minimum 3)
3. Score each option against all 6 decision quality gates
4. Seek outside input (advisors from NETWORK.md, board)
5. Sleep on it (minimum 24 hours for major decisions)
6. Make the decision and document thoroughly
7. Set a review date

### Type 2 — Reversible (two-way door)
- Easy to undo or course-correct
- Lower stakes
- Require speed over thoroughness
- Should be delegated when possible

**Examples:**
- Campaign strategy changes
- Pricing experiments
- Content calendar adjustments
- Vendor selection
- Process changes
- Feature prioritization
- Tool selection

**Process:**
1. Apply the 70% rule — if you have 70% confidence, decide now
2. Document the decision (briefly)
3. Set a review date (2-4 weeks)
4. Course-correct based on results
5. The cost of slow decision-making exceeds the cost of being wrong

## The 70% Rule

> "If you wait for 90% of the information to make a decision, in most cases, you're probably being slow." — Jeff Bezos

- 70% confidence is enough for Type 2 decisions
- Identify the 1-2 data points that would move you from 70% to 90%
- Ask: "Is the cost of getting that data worth the delay?"
- Usually the answer is no — decide and iterate

## RAPID Framework (Bain & Company)

For decisions involving multiple stakeholders:

| Role | Definition |
|------|-----------|
| **R**ecommend | Gathers data, analyzes options, proposes a recommendation |
| **A**gree | Must agree with the recommendation (has veto power) |
| **P**erform | Executes the decision after it's made |
| **I**nput | Provides relevant information and perspective |
| **D**ecide | Makes the final call |

**When to use RAPID:**
- Decisions involving 3+ people
- Cross-functional decisions
- Decisions where accountability is unclear
- Recurring decisions that need a clear process

**FOUNDER:OS mapping:**
- D (Decide) = Founder (for Type 1) or Delegated owner (for Type 2)
- R (Recommend) = Relevant OS framework analysis
- I (Input) = Connected OS frameworks providing data
- P (Perform) = Delegated OS framework or team member
- A (Agree) = Board (for fundraising, major pivots)

## 10/10/10 Framework (Suzy Welch)

For decisions where emotion clouds judgment:

1. How will I feel about this decision in **10 minutes**?
2. How will I feel about this decision in **10 months**?
3. How will I feel about this decision in **10 years**?

Use when:
- Anxiety about a confrontation (firing, tough feedback)
- FOMO about an opportunity
- Sunk cost making you hold onto something
- Personal emotion conflicting with business logic

## Pre-mortem

Before making a major decision, imagine it failed:

1. "It's 12 months from now. We made this decision and it failed completely."
2. "What went wrong?"
3. List all plausible reasons for failure
4. For each reason: is this preventable? What would prevent it?
5. Update the decision with mitigations for top failure modes

## Decision Quality Scoring

After a decision outcome is known (at review date), score the decision:

| Score | Meaning |
|-------|---------|
| **Good decision, good outcome** | System working — keep doing this |
| **Good decision, bad outcome** | Bad luck or new information — don't change the process |
| **Bad decision, good outcome** | Got lucky — improve the process |
| **Bad decision, bad outcome** | Process failed — learn and fix |

The goal is to improve the process, not just the outcomes. Good processes produce good outcomes over time.

## When to Seek Outside Input

Always seek input for:
- Decisions you haven't made before
- Decisions with >$100K impact
- Decisions affecting >30% of the team
- Decisions you're emotionally invested in
- Decisions where you're the only one who thinks it's a good idea

Check NETWORK.md for advisors with relevant expertise.

## Common Decision Biases to Watch

| Bias | Watch for | Counter |
|------|----------|---------|
| Sunk cost | "We've already invested so much" | Only look at future value |
| Confirmation | Seeking data that supports your preference | Assign someone to argue the opposite |
| Recency | Overweighting recent events | Look at 6+ month trends |
| Anchoring | First number frames everything | Generate alternatives independently |
| FOMO | "If we don't do this now..." | Apply 10/10/10 framework |
| Founder ego | "I know best" | Ask advisors, check LEARNINGS.md for past mistakes |
